Based on a review of the AOT codegen, the safepoint ingestion, and the NewGC major collection code, **the issue is almost certainly LLVM dropping or misordering the AOT spill stores, not a bug in the GC's rewrite logic.**

Here is the breakdown of why the evidence zeroes in on AOT LLVM codegen:

### 1. The GC is completely blind to the "argument vs non-argument" distinction
In `heap.rs`, both minor and major collections (via `collect_full`) call `snapshot_roots()`. This function builds a unified `Vec<*const Word>` by seamlessly appending the thread-local `ROOT_STACK`, the JIT maps, and `snapshot_active_aot_roots()`. 

When the major collection visits these roots:
```rust
fn visit_roots(evac: &mut newgc_core::page_heap::PageEvacuator<'_, DylanLayout>, roots: &[*const Word]) {
    for &slot in roots.iter() {
        unsafe {
            let ngc_slot = slot as *mut newgc_core::Word;
            evac.visit(&mut *ngc_slot);
        }
    }
}
```
The GC does not know where the pointer in the `Vec` came from. It blindly rewrites the memory at the given `slot` address to the forwarded location. If the slot’s address was handed to `snapshot_active_aot_roots`, the GC will successfully update it.

### 2. Why does passing it as an argument suddenly fix it?
If `t4` is an argument to an allocating builtin like `%make` (which lowers to `nod_make`), the value is passed directly to the callee. 
If we look at `nod_make` in `src/nod-runtime/src/make.rs`, the very first thing it does is dynamically push its arguments into the thread-local `ROOT_STACK` via `RootGuard`:
```rust
    let mut name_words: [Word; MAKE_MAX_KW_PAIRS] = ...;
    // ...
    let _name_guards: Vec<crate::make::RootGuard> = name_words
        .iter()
        .take(kw_count)
        .map(crate::make::RootGuard::new) // calls register_root on the local stack address
        .collect();
```
When `t4` is passed as an argument, it gains a **second, perfectly reliable RootGuard slot** inside the callee's execution frame. Even if the AOT compiler completely botched the caller's safepoint spill slot, the GC finds the object through the callee's `ROOT_STACK` entry, moves it, and updates the callee's local state.

### 3. The LLVM "Dead Store" / Alias hazard for AOT Safepoints
If `t4` is **not** passed as an argument, its only lifeline to the GC is the `store` LLVM instruction generated by `begin_safepoint` in `codegen.rs`:
```rust
for (i, t) in roots.iter().enumerate() {
    let cur = self.temp_val(*t);
    let slot = self.rent_safepoint_slot(i)?;
    self.builder.build_store(slot, cur).map_err(map_err)?;
}
```
The AOT runtime requires LLVM to spill `cur` into `slot`, call the opaque Dylan function (where the GC runs), and then reload `cur` from `slot`. 

However, `nod_aot_begin_safepoint` captures `slot_base_ptr` into a Rust `thread_local`. If LLVM's `MemorySSA` (or LTO) analyzes `nod_aot_begin_safepoint` and determines the opaque Dylan call doesn't visibly write to that `thread_local` memory, it may interpret the setup as a **Dead Store** or perform **Store-to-Load Forwarding**.

If LLVM optimizes away the `store` or bypasses the `reload` (ignoring the memory rewrite caused by the GC inside the opaque call), the caller will resume using the stale, pre-evacuation SSA value of `t4`.
